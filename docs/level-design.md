# Level 设计与代码层归属

> 版本：v0.2
> 日期：2026-06-14
> 背景：基于 2026-06-13 ~ 2026-06-14 讨论整理，**修正 spec.md v0.5 的核心假设**
> v0.1 → v0.2 主要变化：明确 **L1 = code 注释、L2 = code 实现** 的统一模型；边作为节点方法（Class化）；位置信息移除模型；L3 扩展为 Class 方法 + 外部调用

---

## 一、核心命题：Level 1 和 Level 2 是 graph 从模糊到明确的两个阶段

### Level 1
- 信息不完整、关系不明确时使用
- 用**自然语言**描述
- 人和 AI 都能理解、都能参与
- 目的：**通过对话把想法完善**

### Level 2
- 想法明确后使用
- **AI 主动**把自然语言转换为代码
- 在严密逻辑上锁定关系
- 目的：**让关系精确化、可执行**

### 关键认知
- **不是两种并行的表达方式**
- 是**同一条 graph 在不同成熟度的形态**
- Level 2 是 Level 1 经人 AI 协作后的产物
- Level 1 不消失——它是 Level 2 的来源和对照

### 三个维度对比

| 维度 | Level 1 | Level 2 |
|------|---------|---------|
| **主导者** | 人（描述需求） | AI（转换实现） |
| **介质** | 自然语言（双向沟通） | 代码（单向产出） |
| **目的** | 完善（迭代想法） | 严密化（锁定逻辑） |

### 时序
- 先 Level 1 后 Level 2
- Level 1 改了 → Level 2 标记为 stale → 决定是否重新生成
- 后续如果 Level 2 的逻辑错了，回到 Level 1 改自然语言，再让 AI 重新转换

## 二、为什么用代码作为 graph 介质

### 核心命题
graph 的本质包含**过程性知识**——"怎么做"，不只是"是什么"。过程性知识只能用代码表达，无法用结构化数据（JSON）穷举。

### 三种语义统一在代码

| graph 语义 | 代码实现 |
|-----------|---------|
| 节点存在 | Class 声明 |
| 节点状态 | Class 字段（类型开放——数字、字符串、模糊描述、嵌套对象，因为现实状态本身就是多样的） |
| 节点间关系 | Class 方法（读 source 状态 → 计算 → 写 target 状态） |

代码把这三件事统合在一种介质里。Class 是它们的语义单元。

### 为什么不是 JSON

graph 表达的关系有两层语义：

- **结构层**（"A 和 B 有联系"）—— 数据可表达
- **过程层**（"A 在 X 状态下让 B 进入 Y 状态"）—— **必须代码**

模糊描述（字符串"增加"）、条件分支（if-else）、非线性（阈值切换）、多属性联动（一次改 target 的多个字段）——都是过程性。JSON 要表达这些，要么做不到，要么必须预定义 schema 把开放性封死。

### 开放性是本质

代码享有编程语言的全部表达力（图灵完备）。graph 的关系可以任意复杂、任意精确、任意模糊。

**JSON 是"先设计 schema 再塞数据"；代码是"写任意逻辑，schema 隐含其中"**。

后者才是 graph 关系需要的——因为现实系统的关系本来就没办法被任何固定 schema 穷举。这就是用代码作为 graph 介质的根本理由：**让 graph 拥有编程语言的全部表达力，而不是被数据结构约束**。

### L1 = code 注释，L2 = code 实现（v0.2 核心统一）

前面三种语义（节点存在、节点状态、节点关系）都在 code 里。但 L1（自然语言描述）也在 code 里——**作为注释/docstring**。

这不是另一种存储方式，而是说 **L1 和 L2 是同一段 code 的两个层次**：

```javascript
class population {
  /**
   * 城市常住人口，受出生、死亡、移民影响
   */
  current = 1000
  birthRate = 0.02

  /**
   * 人口越多，住房需求越大，比例约 0.3，超过 1000 时变为 0.4
   */
  static outputs = [{ id: 'housing_demand', target: 'housing', target_port: 'demand' }]
  housing_demand(inputs) {
    const factor = this.current > 1000 ? 0.4 : 0.3
    return this.current * factor
  }
}
```

- 注释（`/** ... */`）= L1，人写、AI 读
- 方法体（`housing_demand(inputs) { ... }`）= L2，AI 写、引擎跑
- **同一份代码，两种角色**

类比 Python：先写 docstring 规划，再填实现。L1 不是"另一种表达"，就是**代码写到一半的状态**。

**这把数据模型简化了**：

| v0.1 模型 | v0.2 模型 |
|----------|----------|
| `node = { description, code }` | `node = { code }`（注释在 code 里） |
| L1 字段 + L2 字段，需要同步 | 单一源，无需同步 |
| 导入导出要支持两种格式 | 只有 code 一种格式 |

**边作为节点方法（Class化）**：边不是独立数据结构，是 source 节点 Class 的方法。方法上方的注释是边的 L1，方法体是边的 L2，`static outputs` 声明拓扑（这条边连到哪个节点的哪个端口）。

## 三、代码层的归属：AI 的内部产物（颠覆性认知）

### 之前的错误假设（spec.md v0.5）

spec.md v0.5 假设代码层是**人和 AI 的公共协议**：
- 双向同步（人改 UI 字段 → 同步 code；AI 改 code → 同步 UI）
- 代码面板（人能看、能改 code）
- 节点 code 字段是人和 AI 共同的操作面

### 新理解

按 Level 1/2 的工作流，**代码层是 AI 的内部产物**：
- 人不写 code
- 人不看 code
- code 是 AI 把 Level 1 翻译为 Level 2 的产物

类比编译器：人写 C，编译器生成机器码，人不看机器码。同理，人写 Level 1，AI 生成 Level 2，人不看 code。

### 真正的人和 AI 的协议

不是 code，是 **Level 1 的自然语言 + URL hash**：

- 人写 Level 1（自然语言描述节点和边）
- AI 读 Level 1，转换生成 Level 2
- URL hash 编码 Level 1（+ 可选 Level 2）

人和 AI 沟通用自然语言，不用代码。

### 这意味着什么

之前做的"代码面板"、"双向同步机制"基础假设错了。

### UI 是 code 的注释视图（v0.2 补充）

L1 在 code 里以注释存在，所以"AI 内部产物"不是黑盒——**人通过 UI 编辑注释，等价于改 L1**。

- 人 UI 编辑"描述" → 实际是改 code 里对应的注释
- AI 读 code → 同时看到 L1（注释）和 L2（实现）
- UI 隐藏 method body，只暴露注释 → 人不需要看 code 也能改 L1

类比 IDE 折叠代码：同一份代码，不同展示层。人看注释视图，AI 看完整 code。

## 四、对当前实现的影响

### 应该去掉或改的

| 当前实现 | 应该改为 | 原因 |
|---------|---------|------|
| 代码面板（人看/改 code） | 去掉，或藏到 debug 模式 | 人不看 code |
| UI ↔ code 双向同步 | 单向：UI 编辑注释（L1），AI 转换实现（L2） | code 是 AI 内部产物 |
| "导出给 AI" 输出 code | 输出 code（含注释），AI 读注释后填实现 | code 是 AI 的输入 |
| 节点 code 字段（人可编辑） | code 是 AI 内部字段；UI 只编辑其中的注释 | 同上 |
| syncNodeCode（每次 UI 改动都重生成 code） | 改为局部修改：UI 改注释只动注释部分 | code 不应该是 UI 改动的产物 |
| 节点 `description` 字段 | 删除（在 code 注释里） | 单一源 |
| 节点 `x, y` 字段（在 JSON 导出里） | 删除（位置是视图信息，不是模型） | 模型/视图分离 |
| 节点 `visual` 字段 | 删除（同上） | 同上 |
| `state.edges` 独立数组 | 删除（边作为节点方法，运行时派生） | Class化 |

### 应该保留的

| 实现 | 原因 |
|------|------|
| 执行引擎（propagate / stepAll） | Level 2 的 code 要跑 |
| URL hash 编解码 | code 的载体（含注释） |
| code 里的注释/docstring | **Level 1 的核心**（人写、AI 读） |
| importJSON 的 code 字段处理 | AI 通过 URL 写入 code 时仍需要 |
| `static outputs` 声明 | 拓扑信息（哪条边连到哪） |

### Level 1 注释的新地位

之前 Level 1 的边 description 是装饰（hover 显示）。

新理解：**code 里的注释是 AI 转换的核心输入**——AI 读注释，在下面填方法体。

- 弱注释："人口 → 住房"。AI 只能猜。
- 强注释："人口越多，住房需求越大，比例约 0.3，超过 1000 时变为 0.4"。AI 能直接转。

UI 应该鼓励详细写注释。可能加引导（"详细描述关系，AI 会转换为代码"）。

### Level 1 节点注释也变重要

不只是边对应的注释，**节点字段上方的注释也变重要**——AI 需要理解节点的语义才能生成正确的 Level 2。

比如"人口"节点的字段注释："城市常住人口，受出生、死亡、移民影响"。AI 读这个，知道要为人口加出生率、死亡率、移民等属性。

UI 应该鼓励详细的节点字段注释。

## 五、人怎么调试 AI 的转换

### 核心问题
如果人不看 code，怎么知道 AI 转的对不对？

### 答案：看执行结果，不看 code

如果 propagate 后结果不对：
- **不是去改 code**
- 而是回到 Level 1 改自然语言描述
- 让 AI 重新转换

人通过自然语言和效果来调试。这要求**执行结果的可视化必须清晰**（节点状态变化一眼能看出来）。

### 与传统"代码调试"对比

| 传统调试 | 新方式 |
|---------|--------|
| 人看代码 | 人看节点状态变化 |
| 加 print / 断点 | 改变 propagate 的输入看输出 |
| 跟踪变量 | 对比预期值和实际值 |
| 改代码修 bug | 改 Level 1 注释，让 AI 重转 |

更高级的抽象——人不需要理解 AI 的实现细节。

### Stale 检测：注释和实现不匹配

人改了注释但实现没跟上 = stale。AI 读 code 一眼能看出：
- 注释说"比例 0.3"，实现写 `* 0.4` → stale
- 注释加了"超过 1000 切换"，实现没有 if 分支 → stale

UI 可以加 stale 标记（节点/方法旁边的指示），让人知道哪里需要 AI 重转。AI 看到 stale 标记时优先重转这部分。

### 对 UI 的要求
- 节点状态变化要醒目（颜色、动画）
- 数值变化要清晰显示
- 可能加"执行历史"视图（每个 tick 的状态对比）

## 六、AI 转换的工作流

### 多轮协作
AI 转换不是一键魔法，是**对话式协作**：

1. 人写 Level 1："人口越多，住房需求越大"
2. AI 转 Level 2：`housing.demand = population.current * 0.3`
3. 人看效果，说"系数太大"
4. AI 改：`housing.demand = population.current * 0.2`
5. 直到满意

### Web as Skill 模式下的多轮

通过 URL hash 多轮交换：
- AI 输出含 Level 2 的 URL
- 用户在 UI 改 Level 1 → 复制新 URL 给 AI
- AI 重新转换 → 输出新 URL
- 用户继续看效果，循环

这是 Git 式的协作：URL 即 commit，每次修改都是新版本。

## 七、设计点（部分已确定，部分开放）

### Q1: AI 转换是全量还是增量？【已定：默认全量 + 锁定】

- 默认全量：每次注释改了，AI 重新生成相关方法体
- 允许"锁定"某段（见 Q2）

### Q2: 人能不能"锁定" Level 2？【开放】

- 即使注释改了，某段实现也不被覆盖
- 用法：人对某段实现满意了，不想 AI 再动

实现：方法上加 `@locked` 注解（JSDoc tag），AI 读到就跳过。

### Q3: AI 转换出 bug 怎么办？【已定：回 L1 重转 + debug 模式】

- 默认回 Level 1 改注释，让 AI 重转
- debug 模式（隐藏入口）看 code，手动改

### Q4: AI 怎么知道什么时候转？【已定：手动触发 + AI 建议】

- 默认手动触发：用户在 UI 点"请 AI 转 L2"
- AI 可以建议：检测到注释详细且实现为空/stale 时提示

### Q5: Level 1 和 Level 2 同时存在一个 graph 里吗？【已定：方法级粒度】

- 粒度是**方法级**（不是节点级，更不是 graph 级）
- 一个节点里：方法 A 是 L2（已转），方法 B 还是 L1（只有注释、方法体空）
- 整个 graph 是"完成度递进"的混合状态

## 十、关键术语表

| 术语 | 含义 |
|------|------|
| **Level 1** | code 的初始状态，只有注释/docstring，方法体空或 stub |
| **Level 2** | code 的成熟状态，注释 + 完整方法体实现 |
| **Level 3** | code 含被外部调用的方法（含但不限于 `tick(dt)`）的演化阶段 |
| **过程性知识** | "怎么做"的知识，只能用代码表达 |
| **结构化数据（JSON）** | "是什么"的数据，无法表达过程性 |
| **代码层** | graph 的 JS Class 表示，是 AI 的内部产物 |
| **L1 = code 注释** | 自然语言描述作为 code 里的注释/docstring 存在（v0.2 核心） |
| **Class化** | 边作为 source 节点 Class 的方法，不作为独立数据结构 |
| **真正的协议** | code（含注释） + URL hash |
| **AI 转换** | AI 读 L1 注释，在下面填 L2 实现的过程 |
| **调试** | 人看执行结果，不看 code |
| **stale** | 注释改了但实现没跟上，需要 AI 重转 |
| **模型/视图分离** | 模型（code）只含语义；视图（位置、颜色）独立，不入 JSON |
