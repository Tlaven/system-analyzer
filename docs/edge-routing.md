# 边路由：布线与端口系统

> 这是**路由层**的设计文档，承接 `visualization-modes.md`（渲染层）、承接 `layouts.md`（布局层）。
> 讲：边怎么从源节点走到目标节点，节点之间如何避让，端口如何分配。

## 1. 三条设计原则

所有边布线决策都从这三条原则推导：

1. **边垂直于节点矩形进出**——端口在矩形侧边中点，切线指向对面端口。人眼沿着边的出/入方向能直接预判它的去向，没有斜射入矩形的突兀感。
2. **避免强烈弯折，曲率整体相近**——不人为制造平顶、急弯或 U 形回折。曲线应该是连贯的、可以一眼追到的，而不是为了"绕过"而牺牲流畅性。
3. **边不被中间矩形遮挡**——人眼可以沿边追踪到目标。如果中间节点挡住边，要么柔和绕开，要么降级到不绕（保留形态可读性）。

**接受的两个不完美**（不试图解决，靠交互吸收）：

- **边与边交叉不管**——半透明叠加让交叉处自然变深，密集区域自然突出；hover 高亮单条边做精确反馈
- **节点四侧都有端口时像蜘蛛**——但单条边路径更短、更直，整体可读性反而更高

## 2. 端口系统

### 2.1 端口的定位：算法层端口，不是数据模型端口

v0.9 重构时把 v0.6/v0.8 时代的"命名端口"（`inputs`/`outputs` 数组实体）砍掉——`io.js` 里 `inputs`/`outputs` getter 永远返回 `[]`，画布不再画端口圆点。这是 v0.9 减法哲学的一部分：边是实例级 `attrs.edges` 数组，端口不需要独立存在。

**重新引入的端口定位**：
- **不入 `attrs.edges[i]`** —— 不破坏 v0.9 实例级 edges 数组的简洁
- **渲染/路由层动态分配** —— 给定当前 `deriveEdges()` 的拓扑，按规则算出每个边的出/入端口位置
- **用户看不到端口编号** —— 端口是几何产物，不是可命名实体，不进 panel / 不进 sourceCode

这与 v0.9 把 edges 从 class 级迁到实例级的"减法"哲学一致：**增加几何复杂度，不增加概念复杂度**。

### 2.2 主方向选侧：四个侧边都可以用

不固定"右出左入"——四个侧边（上/下/左/右）都可以作为出/入侧，由 source 与 target 的相对位置决定。

**选侧规则**：

```
dx = tgt.x - src.x
dy = tgt.y - src.y

if |dx| >= |dy|:  水平占优 → 走左右
  dx >= 0: source 出右、target 入左
  dx < 0:  source 出左、target 入右
if |dx| < |dy|:   垂直占优 → 走上下
  dy >= 0: source 出下、target 入上
  dy < 0:  source 出上、target 入下
```

source 与 target 永远**面对面对齐**——出/入侧正好相对，切线方向一致。

**为何用主方向而非全局方向**：
- 完全隐式，不需要用户配置 `layoutDirection`
- 单条边总是走"最短最直"的那一侧，路径自然短
- 适合拓扑复杂、方向不统一的图

**代价（接受）**：失去"全局主流向感"——但代价由"单条边更短更直"补偿。如果用户需要严格的左→右 / 上→下流向，那是布局层的职责（hierarchical 布局 + LR/TB 方向），不是路由层强制。

### 2.3 整侧统一等分：不按 (source, target) 分组

一个节点的同一侧可能出/入多条边。端口位置的分配规则：

- **整侧统一等分**——同侧所有边（不管对端是谁）在该侧边上等距排开
- **按对端位置排序**——水平侧（左/右）按对端 y 排序，垂直侧（上/下）按对端 x 排序。视觉上端口顺序与对端位置一致，减少交叉
- **不按 (source, target) 分组**——同对多边不再挤在同一位置，而是和其他边一起在整侧均分

```
节点 A 右侧出 3 条边（→B, →C, →D，B/C/D 的 y 各不同）：
  A.right[1/4 高] → B    （B 的 y 最小）
  A.right[2/4 高] → C
  A.right[3/4 高] → D    （D 的 y 最大）
```

如果同对多边（A→B 有 3 条），它们在 A 的右侧也是 3 个不同位置（与其他边一起整侧等分），不再"挤在中间"。

### 2.4 minimal 模式：圆周即端口（不引入显式端口）

minimal 圆形**不画端口圆点**——保持极简姿态。边沿两圆心连线的圆周交点出入：

```
p1 = src + r_src · 单位向量(tgt - src)
p2 = tgt - r_tgt · 单位向量(tgt - src)
```

箭头落在目标圆周上，永远可见；边沿径向出入，自然垂直于圆周。这是 minimal 的"端口"实现——不需要画圆点，圆周本身就是端口。

### 2.5 端口无可见标记（不画圆点）

端口是**抽象的几何概念**——边的出入点位置——不需要可见的圆点标记。用户看到的只有：
- **边本身**——从节点矩形侧边中点出入，自然垂直
- **拖柄**——仅选中节点时显示的 4 个中点圆点，作为"拖出新边"的入口

| | 拖柄（handle）| 端口（port）|
|---|---|---|
| **作用** | 创建边的入口（拖出新边）| 已存在边的出入点（抽象）|
| **可见性** | 仅选中节点时可见 | **不可见**（只存在于几何计算中）|
| **hit 检测** | `hitHandle`（仅 selNode）| `hitPort`（所有节点，命中端口附近）|

不画端口圆点的理由：
- **避免视觉拥挤**——一个节点的同一侧可能出/入多条边，画圆点会让节点边缘布满小点
- **边的轨迹已经表达了出入位置**——曲线从矩形中点垂直出入，用户能直接看到
- **拖柄已经占据"节点边缘圆点"的视觉位置**——再加端口圆点会引起混淆

## 3. 三种布线

### 3.1 straight：不绕（吞吐量模式）

**核心理念**：straight 是"我不在乎单条边"的极简姿态。边从源端口直奔目标端口，**穿过中间节点也不绕**——这是设计意图，不是 bug。

**几何**：直线 `P1 → P2`，`P1/P2` 由端口系统给出。

**何时选 straight**：
- minimal 默认（鸟瞰不在乎单边）
- full 吞吐量模式（信息密度爆表时，curve/orthogonal 都失效）

### 3.2 curve：单段 cubic + bulge 避让

**核心理念**：曲线绕节点像水流绕石头——单段 cubic 在两端切线对齐，中间用控制点同向偏移柔和绕过遮挡。

**算法骨架**：

1. **不分前向/后向**——所有方向统一处理。前向（dx ≥ 0）和反向（dx < 0）走同一套逻辑：端口由主方向法选（见 §2.2），切线由端口侧决定。
2. **端点切线对齐（最高优先级）**：
   - 水平主方向：两端切线水平（source 切线 = (±1, 0)、target 切线 = (±1, 0)，方向一致）
   - 垂直主方向：两端切线垂直（source 切线 = (0, ±1)、target 切线 = (0, ±1)）
   - 这保证 cubic 在端口处垂直于矩形侧边——满足设计原则 1
3. **控制点位置（严格沿外法线，不加任何偏移）**：
   - `cp1 = P1 + dir · t_ctrl`
   - `cp2 = P2 - dir · t_ctrl`
   - `dir` 是端口外法线方向（按主方向 + 相对位置确定）
   - `t_ctrl = clamp(dist · 0.4, 30, 120)`——控制点距离随边长伸缩，让**曲率整体相近**（短边曲率大、长边曲率小，但视觉上"弯度比例"一致）——满足设计原则 2
4. **不避让中间节点（接受遮挡）**：
   - cp 沿外法线方向，**不加任何 bulge 偏移**
   - 中间节点挡住边就挡住——遮挡由半透明叠加 + hover 高亮吸收（§4 交互层处理复杂度，符合"渲染层有上限，复杂度最终靠交互吸收"的核心思想 4）
   - 仅当 **3+ 节点遮挡**时降级为 straight——curve 适合中密度，密度爆表时 straight 才是对的

**为什么不避让（关键取舍）**：

单段 cubic 下，**端点切线严格垂直** 与 **中间避让 bulge** 数学上不可兼得：

```
设 cp1.y = p1.y（端点切线水平），cp2.y = p2.y（同上）
则 cubic 中点 y = 0.125·p1.y + 0.375·cp1.y + 0.375·cp2.y + 0.125·p2.y
             = 0.5·p1.y + 0.5·p2.y
             = (p1.y + p2.y) / 2  ← 锁死，无法偏移避让
```

要让中点 y 可偏移（避让），必须让 cp1.y 或 cp2.y 偏离 p1.y/p2.y——这立即让端点切线倾斜（不再水平），违反"垂直进出"。

**两条出路**：
- **单段 cubic + 放弃避让**（本设计）：端口严格垂直进出，遮挡靠交互吸收
- **多段 cubic + 兼顾两者**：以拼接复杂度 + C2 不连续（视觉弯折）为代价——之前试过，用户反馈"强烈弯折"

本设计选第一条——**端口垂直进出是第一优先级**，避让降级为"靠交互吸收"。

**其他关键取舍**：
- **不分前向/反向**——主方向端口系统让前向/反向天然对称（出右入左 vs 出左入右 几何同构），不需要为反向边单独设计拱形
- **3+ 节点遮挡降级 straight**——curve 不是万能的，密度爆表时降级比强行绕更诚实
- **近距离（dist < 30）控制点收缩到下限**——避免控制点过近导致曲线回卷戳进节点矩形

**minimal 模式降级**：圆心连线加曲线只会变形，没有信息增量——minimal + curve 强制降级为 straight。

### 3.3 orthogonal / polyline：直角通道避让

**核心理念**：直角走线是工程图的语言——Z 形 + 中转通道，可追踪、可数数、好对齐。

**当前实现（polyline）**：
- **前向边**：Z 形（水平出 → 垂直中转 → 水平入），中转段位置由 `findOrthogonalChannel` 找空通道
- **反向边**：U 形外圈（绕到节点上方或下方），保持"反向"视觉语义
- **同行**：直接水平线

**避让比 curve 容易**：直角天然走"节点之间的空隙"——曲线要算曲率，折线只要找空列。

**暂不重做**：当前 polyline 已经稳定，主方向端口系统主要服务 curve。polyline 的"右出左入"硬编码与 curve 的"主方向"暂不强行统一——切边样式时端口会重排是可接受的。

## 4. 边交互（curve 专有）

### 4.1 半透明叠加

**核心理念**：curve 模式下边默认半透明（alpha ≈ 0.55），交叉处自然叠加变深。

**视觉效果**：
- 边稀疏区域：浅淡，不抢眼
- 边密集交叉区域：自然变深，**密集区域自然突出**——不需要任何额外计算
- 选中/hover 边：alpha = 1.0，与底色形成强对比

**为什么 curve 专有**：
- straight 本就是"不在乎单边"的姿态，半透明无意义
- polyline 的直角交叉处叠加会产生混淆（两条直角线交叉看起来像 4 段），半透明反而更乱
- curve 的柔和曲线交叉处叠加视觉上像"两根曲线交叉"，符合直觉

### 4.2 hover 高亮

**核心理念**：鼠标移到边上变亮或变色，提供精确反馈。

**实现**：
- `hitEdge` 在 curve 模式下用 cubic bezier 采样距离（20 段线性近似），不再用直线距离
- `state.hoverEdge` 存当前 hover 的边 id（已存在，用于 tooltip）
- 渲染时若 `state.hoverEdge === e.id`：边颜色变 `es.sel`（选中色）、alpha = 1、lineWidth 加粗

**配合半透明**：默认半透明 → hover 全实，对比强烈，反馈清晰。

## 5. 9 组合矩阵（minimal/medium/full × straight/curve/polyline）

| | straight | curve | polyline |
|---|---|---|---|
| **minimal** | ✓ 默认（鸟瞰拓扑）| ✗ 降级为直线（圆心连线加曲线只会变形）| △ 可选（圆形间直角折线，仍简洁）|
| **medium** | ✓ 快速编辑 | ✓ 默认（追踪友好 + 半透明叠加）| ✓ 工程化排版 |
| **full** | ✓ 信息吞吐 | ✓ 单边追踪 + 调试 | ✓ 调试对齐（首选）|

**组合语义**：
- **minimal + straight**：鸟瞰拓扑，边穿过节点没关系，hover 补可见性
- **minimal + curve**：强制降级为 straight（圆心连线加曲线没意义）
- **medium + curve**：默认——柔和可追踪 + 半透明叠加让密集区域自然突出
- **full + polyline**：调试对齐首选——直角通道配合 grid 布局

## 6. 与现状的差异

当前实现状态（`renderer.js` / `utils.js`）：

| 项 | 现状 | 目标状态（本次重做） |
|---|---|---|
| 端口选侧 | 固定右出左入 | 主方向法选侧（4 侧）—— **仅 curve 模式** |
| 端口分配 | 半固定 (source, target) 分组 + 组内等分 | 整侧统一等分（按对端位置排序）—— **仅 curve 模式** |
| straight / polyline 端口 | 固定右出左入 | 保持不变 |
| curve 几何 | 前向/反向分流 + lead + 2 段 cubic + 反向拱形 | 单段 cubic，端点切线严格沿外法线 |
| curve 避让 | 法向偏移避让 | **不避让**（端点切线严格垂直优先于避让，遮挡靠交互吸收）|
| curve 遮挡降级 | 仅反向 U 形 / 前向 2 段 cubic | 3+ 节点遮挡降级 straight |
| 多边同对 | 端口分组让同对挤在中间 | 整侧等分让同对也分散 |
| 端口可见标记 | medium/full 画端口圆点 | **不画**（端口是抽象概念，圆点拥挤且与拖柄混淆）|
| 边 hit 检测 | 直线 `distSeg` | curve 模式用 cubic 采样 |
| 边视觉 | 实心渲染 | curve 模式默认半透明 + hover 高亮 |
| hoverEdge | 仅 tooltip | tooltip + 渲染高亮 |

## 7. 落地次序

1. **端口主方向 + 整侧等分**（仅 curve 模式）
   - `computeNodePorts` 在 curve 模式下走新规则，其他模式保持旧规则
   - 切边样式时端口重排是可接受的
2. **单段 cubic（端点切线严格垂直）**
   - 提取 `computeCurveGeometry` 公共函数（renderer + hitEdge 共用）
   - 删掉 backwardArch / 2 段 cubic / lead / 前向反向分流
   - cp 沿外法线方向，不加任何 bulge（避让靠交互层吸收）
3. **半透明 + hover 高亮**
   - curve 模式默认 alpha = 0.55
   - hoverEdge 渲染时变色 + alpha = 1
4. **hit 检测改 cubic 采样**
   - curve 模式下 hitEdge 用 cubic bezier 20 段采样

minimal 全程不动（圆周即端口）。所有模式都不画可见端口圆点。

## 8. 不做什么

- **不画边标签**（三档都不画）。边的语义靠 hover tooltip。
- **不引入命名端口**（v0.6/v0.8 那种 `inputs`/`outputs` 数组实体）。端口只作算法层布线辅助。
- **不在 minimal 引入端口**。圆周即端口，保持极简姿态。
- **不暴露端口编号给用户**。端口是几何产物，不进 panel / 不进 sourceCode。
- **不让 straight 绕路**。straight 的"不绕"是设计意图。
- **不强制 minimal + curve**。圆心连线加曲线只会变形，强制降级为 straight。
- **不解决边交叉**。半透明叠加 + hover 高亮已足够，强求解只会引入复杂度。
- **不重做 polyline**。当前 U/Z 形稳定，polyline 不需要主方向端口（保持右出左入）。

## 9. 与其他层的关系

- **承接布局层**（`docs/layouts.md`）：节点的 x/y 位置由布局层决定。端口分配依赖节点的相对位置（主方向法选侧）。布局层如果引入 `layoutDirection` 配置可作为可选 override，但默认仍是几何驱动。
- **承接渲染层**（`docs/visualization-modes.md`）：边的颜色/粗细/箭头由渲染层决定。端口只是几何概念，不画可见标记。
- **不耦合数据模型**：端口化**不**改 `attrs.edges` schema。所有变化都在 `utils.js`（几何）/ `renderer.js`（绘制）/ `input.js`（hit 检测）三层。
