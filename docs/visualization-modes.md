# 节点可视化：三档信息密度

> 这是一份"要做成什么样子"的意图文档，不是实现指南。
> 写给未来的我（或者 AI 协作者），用来理解三档模式的设计取舍。
>
> 配套文档：
> - `docs/edge-routing.md` — 路由层（边布线 + 端口系统 + 避让算法）
> - `docs/layouts.md` — 布局层（节点位置算法）

## 1. 问题背景

画布上的节点不能"什么都显示"——同一张图，**用户在不同时刻关心的颗粒度不同**：

- 缩到能看 50 个节点时，只关心"哪里有节点、是什么类"，节点编号、属性都是噪声
- 正常编辑/连线时，关心"这是哪个实例、关键参数是多少"
- debug 时，关心"这个节点的描述说了什么、属性全列出来"

如果用一套渲染规则覆盖所有场景，结果就是要么挤、要么空。
v0.6 时代其实已经有 `infoLevel` 配置，但 medium / full 几乎一样、description 从不画在画布上——
本质上是两档（极简 / 详细），第三档是装饰。这次重写把三档的差异讲清楚。

---

## 2. 三档定位

| 档位 | 心智用途 | 切它的时机 |
|------|---------|-----------|
| **minimal** | 鸟瞰结构 | 缩到看拓扑形态、不在意具体值 |
| **medium** | 工作编辑 | 正常连线、调属性 |
| **full** | 检视 debug | 排查数据流、对图细节 |

每一档都为这个用途服务，**不是"少一些属性 / 多一些属性"的渐进**，而是不同显示策略。

---

## 3. 三档显示规则

### 3.1 内容矩阵

| 元素 | minimal | medium | full |
|------|:---:|:---:|:---:|
| 节点形状 | **圆形** | 圆角矩形 | 圆角矩形 |
| 节点名称（主标题）| ✓ | ✓ | ✓ |
| varName 副标题 | ✗ | ✗ | ✓ |
| 节点属性 | ✗ | ✓ | ✓ |
| 端口圆圈（节点边缘出入点）| ✗（圆周即端口）| △ 重新引入作布线辅助 | △ 重新引入作布线辅助 |
| 画布内 description | ✗ | ✗ | ✓ |
| 边标签 | ✗ | ✗ | ✗（三档全无）|
| 节点 hover tooltip | ✓ | ✓ | ✓ |
| 边 hover tooltip | ✓ | ✓ | ✓ |

**端口状态说明**：v0.9 重构时把 v0.8 时代的命名端口（inputs/outputs 数组实体）砍掉——画布不再画端口圆点，`inputs`/`outputs` getter 永远返回空数组。后续在路由层重新引入**算法层端口**（不入 attrs，仅作布线辅助），详见 `docs/edge-routing.md`。当前状态：medium/full 暂用"右中点 → 左中点"硬编码出线，待端口化替换。

### 3.2 关键决定

- **节点名称** = `attrs.name`（v0.9 起是实例级 class field，class 默认 + 实例 override），留空回退 `cls.name`，再留空回退 `varName`。这是节点身份。
- **varName 副标题** 仅在 full 显示。minimal 不需要（圆形里塞不下两行）；medium 是工作模式，varName 的 `<ClassName>_<n>` 后缀噪声大，主标题名称已经够定位实例。full 才需要副标题，因为 debug 时 varName 是程序里引用实例的"真名"。
- **属性 = 节点属性，不含引用槽**。v0.9 起边是实例级 `attrs.edges` 数组，画布显示为边，不进属性区。属性区展示**合并视图**：`{...cls.attrs, ...inst.attrs}` 减去 `name`、`description`、`edges`。实例没 override 时显示 cls.attrs 默认；实例追加 class 没有的字段也显示。
- **description 两层**（v0.9 起）：`cls.description` 是 class 默认，`inst.attrs.description` 是实例 override。`wrapInstance.description` getter 优先取实例、回退 class。full 把它画在节点矩形内最下方；tooltip 同样取这个 getter。
- **边标签三档都不画**。边的语义靠 hover tooltip 显示，画布上保留线本身和箭头。
- **形状分两种**：minimal 圆形（视觉简洁、可大量并列）；medium/full 圆角矩形（结构化排版）。同一个节点切档时形状会变。
- **端口的重新引入**只作为**布线辅助**，不恢复 v0.6 时代的"命名端口实体"。端口是渲染/路由层的几何产物，不是数据模型——保留 v0.9 的减法哲学：增加几何复杂度，不增加概念复杂度。

## 4. 几何

### 4.1 minimal 圆形

- 半径：`max(22, textWidth/2 + 8)`，封顶 ~40px。短名字得到小圆，长名字自动扩大但有限制。
- 节点名称居中（超长截断加 `…`）。
- **没有显式端口**。圆周即"端口"，任何方向都能出入。

**边端点（v0.9 径向连接）**：
- 数学层面：源圆心 → 目标圆心
- 渲染层面：`p1 = src + r·û`，`p2 = tgt - r·û`，其中 `û` 是源→目标单位向量
- 边沿径向出入，**自然垂直于圆周**——既不像早期版本那样被圆覆盖，也不像固定方位点那样在多目标时撞角度
- **箭头落在目标圆周交点 `p2`**，永远可见

为什么不固定边出口在某个圆周方位？因为同一个源节点可能有多条边指向不同目标，
强行在圆周上分配"出口位"既要算扇区又会重叠，不如让所有边共享圆心起点，
方向自然由目标位置决定——视觉上更干净。

**minimal 不绕路**（核心设计约束）：minimal 模式下，边可能被路径上的**其他**圆形节点遮挡——这是有意的。minimal 是鸟瞰模式，绕路破坏简洁。**遮挡的可见性靠 hover 补**——hover 整条边时高亮完整路径（含被覆盖段）。详见 §8.3。

### 4.2 medium / full 圆角矩形

**节点矩形结构（full 示意）**：

```
┌────────────────────────────┐
│   attrs.name 或 cls.name (主标题)  │  ← 顶部
│   varName (副标题灰)       │  ← 仅 full
│   key1            value1   │
│   key2            value2   │
│   key3            value3   │
│   key4            value4   │
│   wrapInstance.description (斜体灰) │  ← 底部，仅 full
●                          ●  ← 端口嵌在左右边
└────────────────────────────┘
```

medium 比 full 少两行（副标题、底部 description），其余同。

**端口位置（重新引入作布线辅助）**：嵌在矩形左右两条边的中点附近，按端口数等分高度，半圆露在矩形外。和 v0.6 时代的"端口在矩形外悬空"不同——嵌在边上让端口和节点形成一个整体，避免端口"飞出"。

**端口是算法层端口，不是命名端口**（v0.9 设计约束）：
- 不入 `attrs.edges[i]` 字段——保持 v0.9 实例级 edges 数组的简洁
- 渲染时按当前边的拓扑动态分配（详见 `docs/edge-routing.md`）
- 用户**看不到端口编号**——端口是几何产物，不是可命名实体

**边端点**：源端口圆圈 → 目标端口圆圈，箭头画在目标端口圆圈外侧。

**当前 v0.9 状态**：端口砍掉，边端点硬编码为"源右中点 → 目标左中点"，多边同对靠 `idx*12` 偏移端点位置（端口在"飘"）。后续端口化会替换这套硬编码——把端口圆点重新画在矩形左右边，并把 `idx*12` 偏移升级为真端口分配。

## 5. 防溢出（核心设计约束）

> **画布上不允许出现文本超出节点边界的情况。** 这是设计约束，不是 best effort。

### 5.1 节点宽度自适应扩展

- 测量所有可见行的实际宽度（主标题、副标题、每个属性 row `key  value`），
  取最大值 + padding 作为节点宽度。
- 下限：`NODE_MIN_W`。
- 上限：`min(视口世界宽 × 0.4, 600)`——大屏可显示更多，但不超过视口一半避免遮蔽邻居。

### 5.2 值精度与折叠（统一走 `formatScalar(v)`）

属性 row 的右值都走 `formatScalar(v)`：

- `|v| ≥ 1e7` 或 `|v| < 1e-4` → `toExponential(2)`
- 非整数 → `toFixed(2)`
- 整数部分超 12 位 → `toExponential(2)`
- 其他整数 → 原样
- 普通对象 → `{…}`（空对象 → `{}`）
- 数组 → `[len]`（带长度，比纯 `[…]` 更有信息）
- 函数 → `ƒ`
- null/undefined → `null` / 空

数字是最容易撑爆节点宽度的来源，显式精度规则比"截断加 …"更有信息量。
对象/数组不试图在画布上展开——画布是结构图，不是 JSON viewer。

### 5.3 长字符串截断

所有超长文本走 `truncateText(ctx, text, maxW)`：宽度超过可用区时，逐字符砍尾加 `…`。
应用范围：主标题、副标题、属性 key、属性 value。

### 5.4 属性数上限

- medium：最多 4 行 + `… +N` 提示行
- full：最多 6 行 + `… +N`

属性数超过上限不直接静默截断——`… +N` 让用户知道"还有"，主动切 panel 看完整列表。

## 6. Hover Tooltip

### 6.1 节点 hover

显示 `wrapInstance.description`（v0.9 起优先 `inst.attrs.description` 回退 `cls.description`）。已有逻辑（`updateTooltip()`），保留。

为什么 description 默认走 hover 而不是画在节点上？因为同一张图里同 class 的多个节点
共享同一段 cls.description（v0.9 起可加实例级 override 区分），画在每个节点上仍是冗余；用户需要时悬停即可。

### 6.2 边 hover

显示 `edge.description`（来自源实例 `attrs.edges[i].description`）。
`state.hoverEdge` 存边 id 字符串（活过 `runSource` 重建），`updateTooltip()` 渲染时优先显示边描述。

### 6.3 Tooltip 优先级

边 hover > 节点 hover（边更细更难命中，命中时优先展示）。

## 7. 不做什么（避免的范围）

- **不在 minimal 圆形里画属性**。圆形几何不适合多行排版。
- **不在画布上画边标签**。三档统一移除。边的语义靠 hover。
- **不显示引用槽作为属性行**。v0.9 起边是实例级 `attrs.edges` 数组，画布显示为边，不进属性区。
- **不让 description 在 minimal / medium 上画布出现**。minimal 圆形塞不下；medium 是工作模式，画布上多一行就少一行属性。
- **不做"展开对象/数组"的内联 JSON 显示**。画布不是 JSON viewer。
- **不重新引入边方向标记的额外装饰**（除了已有的箭头）。边的方向靠箭头表达，足够。
- **不在 minimal 模式引入端口**。圆周即端口已经够；多边同对在 minimal 用"圆周角度分配"实现并行感，保持极简姿态。
- **不恢复 v0.6 时代的命名端口实体**（inputs/outputs 数组）。端口只作为算法层布线辅助重新引入，不入数据模型。

## 8. 边可视化哲学

边可视化是 minimal/medium/full 三档共有的子问题——任何档位都要画边，但策略不同。

### 8.1 四个核心思想

剥离具体技术，所有边可视化方法共通的核心思想：

1. **空间分离 > 视觉编码**：区分边首选物理位置（不同的出线点、不同的通道、不同的轨道），而不是颜色/透明度/粗细。Port Assignment、Track Routing、Orthogonal Bus 都在做同一件事——把扇出从一个点摊到多个点上。
2. **路径规则化（方向约束）**：给所有边一套统一的方向规则，眼睛才能追踪。Layer 强制拓扑流向（左→右 / 上→下），Orthogonal 强制水平/垂直段。规则越一致，认知负担越低。
3. **分层解耦（布局/路由/渲染三层）**：所有方法本质上都只在某一层工作——**布局（节点放哪）→ 路由（边怎么走）→ 渲染（边怎么画）**。这层认知让你可以独立优化某一层而不推翻其他层。
4. **渲染层有上限，复杂度最终靠交互吸收**：画的时候降复杂度（前三种）有上限，图足够大时失效。真正的复杂度管理发生在交互层——Focus+Context / Filter / Semantic Zoom。

### 8.2 三档 × 三布线 = 9 组合

边布线有三种 `edgeStyle`（详见 `docs/edge-routing.md`）：straight / curve / orthogonal。和三档 infoLevel 组合：

| | straight | curve | orthogonal |
|---|---|---|---|
| **minimal** | ✓ 默认（鸟瞰）| ✗ 降级为直线 | △ 可选（圆形间直角折线仍简洁）|
| **medium** | ✓ 快速编辑 | ✓ 默认（追踪友好）| ✓ 工程化排版 |
| **full** | ✓ 信息吞吐 | ✓ 单边追踪 | ✓ 调试对齐 |

**minimal + curve 强制降级为直线**：圆心连线加曲线只会变形，没有信息增量。

**一句话定位**：
- straight 跨三档都是"不在乎单条边"的极简姿态——minimal 默认、full 吞吐
- curve 是 medium 的灵魂——靠曲率避让，柔和可追踪
- orthogonal 是 full 的利器——靠通道避让，工程级精确

### 8.3 minimal 不绕路（hover 补可见性）

minimal 模式下，边可能被路径上的其他圆形节点遮挡。这是**有意的**——minimal 是鸟瞰模式，绕路破坏简洁。

**遮挡可见性靠 hover 补**：hover 整条边时高亮完整路径（含被覆盖段），用户主动选择看哪条边。本质"承认遮挡、但 hover 可见"，符合 minimal "看形态不看细节"的定位。

**对比 medium/full**：这两档的 curve/orthogonal 必须绕路（详见 `docs/edge-routing.md`），因为它们是"追踪每条边"的工作模式，被覆盖的边等于丢失信息。

## 9. 三层架构（相关文档）

可视化决策按布局/路由/渲染三层组织，每层一份文档：

| 层 | 文档 | 解决的问题 |
|---|---|---|
| **渲染层** | `docs/visualization-modes.md`（本文档）| 画布上画什么（节点显示规则、防溢出、tooltip）|
| **路由层** | `docs/edge-routing.md` | 边怎么从源走到目标（端口系统、避让算法、三种布线）|
| **布局层** | `docs/layouts.md` | 节点应该放在哪里（四种布局 + grid 新增）|

每层独立演化，互不耦合。`config.infoLevel` 控制渲染层，`config.edgeStyle` 控制路由层，`config.layout` 控制布局层。
